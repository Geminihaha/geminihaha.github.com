# Zombie Hunter — 개발 기록

## 개요
뱀파이어 서바이버 스타일의 웹 게임. 순수 HTML + Canvas + JavaScript로 제작됨.

- **원본 출처**: Ghost Hero 레포지토리 내 prototype
- **플레이**: `zombiehunter/index.html` 브라우저 열기
- **빌드 없음**: 정적 HTML+JS, 번들러 불필요

### 2026-07-07 10:58 (약) — 채찍(Whip) 공격 히트 판정 버그 수정
- **증상**: 1번 캐릭터(안토니오) 자동 공격 시 범위 호(arc)는 시각적으로 표시되나, 호 안에 몬스터가 들어가도 데미지를 전혀 주지 못함.
- **원인 분석**: `WEAPON_DEFS.whip.update()` 내 `hit:` 함수에서 거리(`dist < r`) 및 각도(`Math.abs(d) < arc/2+0.2`) 조건이 있었으나, 각도 계산(`d = a - angle`)과 시각적 호 생성 방식(`ctx.arc`) 간 정합성 문제로 피격 판정 누락.
- **수정 사항**:
  - `hit()` 함수 재작성: 거리(90px) 조기 탈출 후, 각도 차이(`a - angle`)를 정규화하여 판정.
  - 진단 도구 적용 및 제거: `return true`(무조건 명중) 테스트, 화면 흔들림(`g.screenshake=12`), 히트 카운터 HUD(`히트:N`), 데미지 팝업 강화(22px 볼드+테두리).
  - 버그 복구: `arc/r` 변수 선언이 삭제되었던 문제 해결.
- **검증**: 모바일 환경 정상 작동 확인, 모든 무기 피격 판정 확인.

### 2026-07-07 18:00 (약) — 메인 메뉴 입력 누락 현상 수정
- **증상**: 메인 화면에서 1초 이상 머무른 후 캐릭터 선택 터치가 동작하지 않음.
- **원인 분석**: `mouse.justClicked` 플래그를 게임 루프 하단에서 매 프레임 초기화하여, 터치 이벤트가 프레임 간극에 발생하면 플래그가 소비되기 전에 리셋됨 (레이스 컨디션). 또한 게임 루프 기반 입력 처리(`updateMenu` 내 hover + justClicked 체크)는 `requestAnimationFrame` 타이밍에 의존하므로 모바일 환경에서 미세한 지연이 발생하면 입력이 유실됨.
- **수정 사항**:
  - 캐릭터 선택 로직을 게임 루프(`updateMenu()`)에서 **완전히 분리**하여 이벤트 핸들러(`touchstart`, `mousedown`, `click`)에서 직접 처리하도록 변경.
  - `handleMenuClick()` 함수 추가: 터치/클릭 좌표를 캔버스 좌표로 변환한 후, 캐릭터 목록 상자와 비교하여 일치하면 즉시 `startGame()` 호출.
  - 게임 루프의 전역 `mouse.justClicked=false` 제거 (중복/조기 리셋 방지).
- **결과**: 터치 이벤트 발생과 동시에 메뉴 선택이 처리되므로, 프레임 타이밍이나 1초 이상 대기 후에도 항상 정상 동작.

### 2026-07-07 19:00 (약) — 리사이즈 이벤트 크래시 수정 + 터치 디버그 오버레이

  - 터치 시 위치에 빨간 점 표시 추가 (터치 좌표 정확도 확인용).

### 2026-07-07 20:30 (약) — 터치 이벤트 차단 문제 해결 (근본 원인)
- **증상**: 메뉴 화면 진입 후 터치 좌표가 1초 정도 업데이트되다가 멈춤 → 이후 터치 입력完全 무시.
- **원인 분석**: 모바일 브라우저(Chrome Android)에서 `touchstart`/`touchmove`에 JS `e.preventDefault()`를 호출하면, 브라우저의 스크롤 억제 휴리스틱이 개입하여 일정 시간 후 터치 이벤트 처리 자체를 차단하는 현상 확인.
- **수정 사항**:
  - JS `e.preventDefault()`를 모든 터치 핸들러에서 제거 (passive:true로 변경).
  - CSS `touch-action:none` 속성을 캔버스에 적용하여 브라우저 레벨에서 스크롤/줌 방지.
  - 추가 CSS: `-webkit-touch-callout:none; -webkit-user-select:none;` (.getSelection 방지).
- **검증**: 모바일 환경에서 터치 좌표가 지속적으로 업데이트되고, 메뉴 선택이 정상 동작하는지 확인 필요.

### 2026-07-07 (추가) — 좌표 변환 실패로 인한 좌측 상단(0,0) 이동 버그 수정
- **증상**: 터치/클릭 시 포인터가 항상 좌측 상단 (0,0)으로 이동.
- **원인**: `toCanvasCoords()` 함수(line 328)가 `const canvas` 선언(line 381)보다 먼저 정의됨. `const`는 TDZ(Temporal Dead Zone) 적용으로, 함수 호출 시점에 `canvas`가 `undefined` → `canvas.getBoundingClientRect()`에서 TypeError 발생 → 예외 처리 없이 좌표 (0,0)으로 폴백.
- **수정**: `toCanvasCoords()` 내부에서 `canvas` 대신 `document.getElementById('gc')`를 직접 참조하도록 변경. 함수 호출 시점과 무관하게 안전하게 DOM 요소를 가져옴.
- **검증**: `node --check game.js` 통과. 실제 모바일에서 터치 위치가 정확히 표시되는지 확인 필요.

### 2026-07-07 21:15 (약) — 메인 화면 1초 후 전체 입력 먹통 현상 수정 (실제 근본 원인)
- **증상**: 메인 화면 진입 1초 정도 후 캐릭터 선택 등 모든 동작이 멈춤.
- **원인 분석**: `loop()` 함수의 1초 간격 디버그 로그(`console.log`)가 `game.player.x`를 널 체크 없이 참조. 메뉴 상태에서는 `game.player=null`이므로 진입 후 첫 로그 타이밍(1초)에 `TypeError` 발생. 이 로그 코드는 `game.update()/render()`를 감싸는 `try/catch` 범위 밖에 있어 예외가 그대로 `requestAnimationFrame` 콜백을 중단시킴 → 다음 프레임 예약이 안 되어 렌더/업데이트 루프 자체가 완전히 정지. 캐릭터 선택 이벤트 핸들러는 별도로 동작해 내부적으로 호출은 되지만 화면이 갱신되지 않아 "아무 반응 없음"으로 보임.
- **수정 사항**: 디버그 로그의 `player=(...)` 부분에 `game && game.player` 널 체크 추가.
- **비고**: 기존 터치 `preventDefault` 이슈와는 별개의 회귀 버그. 디버그 로그/카운터를 추가하면서 노출된 것으로 추정.

### 2026-07-07 21:40 (약) — 인게임 터치/드래그 이동이 항상 좌측 상단으로 쏠리는 버그 수정
- **증상**: 게임 플레이 화면에서 화면을 터치/클릭/드래그하면 터치 위치와 무관하게 캐릭터가 좌측 상단 방향으로 이동.
- **원인 분석**: 렌더링은 `screen = world - camX/camY` 규칙을 쓰고 있고(`camX`는 뷰포트 좌측 상단 모서리의 월드 좌표, 예: `resize()`에서 `camX = player.x - W/2`), 따라서 화면→월드 역변환은 `world = screen + camX`여야 함. 그런데 `updatePlayer()`의 터치 이동 처리(`game.js` 728행)가 `wx = mx + camX - W/2`, `wy = my + camY - H/2`로 계산해 화면 절반 크기만큼 추가로 빼고 있었음 → 화면 정중앙을 터치해도 목표 지점이 뷰포트 좌측 상단 모서리의 월드 좌표가 되어버려 캐릭터가 항상 좌상단으로 이동.
- **수정 사항**: `wx=mx+this.camX, wy=my+this.camY`로 불필요한 `-W/2`, `-H/2` 오프셋 제거.
- **검증**: 화면 우측-중앙(예: `mx=W, my=H/2`)을 터치할 때 목표 월드 좌표가 플레이어 기준 오른쪽(`dx=+W/2, dy=0`)으로 정확히 계산되는 것을 Node 스크립트로 확인.

### 2026-07-07 22:10 (약) — 가상 스틱(virtual joystick) 방식 조작으로 변경
- **개요**: 기존에는 터치/클릭한 화면상의 절대 좌표(월드 좌표로 환산)를 향해 캐릭터가 계속 이동하는 "탭-투-무브" 방식이었음. 이를 터치/클릭 시작 지점을 기준점(스틱 베이스)으로 고정하고, 그 지점 대비 현재 포인터 위치의 방향으로 이동하는 가상 스틱 방식으로 변경.
- **주요 수정**:
  - `joystick={active,baseX,baseY,curX,curY}` 전역 상태 추가.
  - `mousedown`/`touchstart` 시 `state==='playing'`이면 해당 좌표를 스틱 베이스로 고정하고 활성화, `mouseup`/`touchend` 시 비활성화.
  - `mousemove`/`touchmove`는 스틱이 활성화된 동안 현재 포인터 좌표만 갱신 (베이스는 고정).
  - `updatePlayer()`: 절대 월드좌표 목표 방식 제거, 스틱 베이스→현재 포인터 방향 벡터(데드존 8px)로 이동 방향 결정.
  - `renderHUD()`: 플레이 중 스틱이 활성화된 동안 베이스 위치에 반투명 외곽 원, 방향에 따라 최대 반경(50px) 내에서 움직이는 내부 스틱 원을 화면 좌표에 그려 시각적 피드백 제공.
- **비고**: 스틱은 화면 좌표 기준(카메라 이동과 무관)이며, 메뉴/레벨업 화면에는 영향 없음(`state==='playing'`일 때만 동작).

### 2026-07-07 22:40 (약) — 번개링/화염봉 공격이 체력 높은 몬스터에만 집중되고 약한 몬스터가 근처에 방치되는 문제 수정
- **증상**: 번개링/화염봉을 쓰는 캐릭터로 플레이 시, 공격이 체력(에너지) 많은 몬스터 쪽에 계속 쏠리고 체력 낮은 몬스터는 공격을 받지 않은 채 캐릭터 근처에 계속 몰려드는 현상.
- **원인 분석**:
  - `lightningRing`(번개링): 사거리 내 몬스터 목록을 만든 뒤 `choice()`로 완전 무작위 대상을 선택. 무작위 선택은 체력과 무관하지만, 체력 낮은 몬스터는 한 번 맞으면 즉시 죽어 목록에서 빠지는 반면 체력 높은(탱키) 몬스터는 여러 번 맞아도 살아남아 계속 목록에 남음 → 결과적으로 "맞는 모습이 계속 보이는" 대상이 체력 높은 쪽으로 편중되어 보이고, 새로 접근하는 약한 몬스터들은 매번 낮은 확률(1/N)로만 선택되어 방치되기 쉬움.
  - `fireWand`(화염봉): 몬스터를 전혀 조준하지 않고 캐릭터의 마지막 이동 방향(`p.fx,p.fy`)으로만 직선 발사. 캐릭터가 멈추거나 방향을 틀면 그 방향에 몬스터가 없어도 계속 허공에 발사되어, 정작 근처에서 둘러싸는 몬스터(체력과 무관하게)는 전혀 맞지 않고 캐릭터 옆에 방치됨.
- **수정 사항**:
  - `lightningRing`: 무작위 선택(`choice`) 대신 사거리 내 **가장 가까운(=가장 위협적인) 몬스터**를 우선 타격하도록 변경. 항상 근접한 몬스터부터 처치되므로 약한 몬스터가 방치되어 쌓이지 않음.
  - `fireWand`: `magicWand`(마법봉)와 동일하게 **가장 가까운 몬스터를 조준**하여 발사하도록 변경(사거리 350*areaMul, 대상 없으면 기존처럼 이동 방향으로 발사).
  - 더 이상 쓰이지 않는 `choice()` 유틸 함수 제거.
- **검증**: Node 스크립트로 가까운 약한 몬스터(hp:20, dist:30)와 먼 탱키 몬스터(hp:500, dist:150)가 섞여 있을 때 가까운 쪽이 선택되는지 확인. 실제 모바일/브라우저에서 다수의 몬스터가 몰렸을 때 체감 확인 필요.

### 2026-07-08 (약) — 번개링이 몬스터 hp가 낮아지면 아예 공격하지 않는 문제 수정 (진짜 근본 원인)
- **증상**: 번개링 사용 시 몬스터 체력이 낮아지면(=거의 죽으면) 이후로는 그 몬스터를 전혀 공격하지 않는 것처럼 보임. 위 22:40 수정(가까운 몬스터 우선 타격)을 적용한 뒤에도 재현됨.
- **원인 분석**: `garlic`(마늘), `lightningRing`(번개링, 스플래시 포함), `kingBible`(성서), `fireWand`(화염봉 폭발 스플래시)는 모두 `updateWeapons()` 단계에서 `e.hp -= dmg`로 몬스터 체력을 **직접** 깎는다. 그런데 몬스터를 죽이고 나서 `entities` 배열에서 제거하고 킬 카운트/XP 젬 지급을 처리하는 로직은 `checkCollisions()`의 "발사체가 몬스터에 맞음"/"공격(attack) 판정" 두 분기 안에만 존재했음. 즉 위 4개 무기로 마무리타격(hp를 0 이하로 만듦)을 한 경우, 몬스터는 hp<=0 상태로 배열에 계속 남지만 **제거되지 않음**.
  - 이후 모든 무기의 타겟팅 로직은 예외 없이 `e.hp<=0`을 걸러내므로, 이 "죽었지만 안 지워진" 몬스터는 번개링을 포함한 모든 공격에서 영원히 제외된다. 동시에 `updateEntities()`의 이동 로직은 hp를 확인하지 않고 계속 플레이어를 향해 이동시키므로, 화면상으로는 살아있는 좀비가 캐릭터 근처에 멈춰 서서 "공격을 안 받는" 것처럼 보이고 계속 쌓인다(직전 22:40 항목에서 다룬 "약한 몬스터가 근처에 몰리는 현상"의 진짜 원인이기도 함).
- **수정 사항**: `checkCollisions()` 최상단에 `entities`를 역순으로 훑어 `type==='enemy' && hp<=0`인 개체를 찾아 킬 카운트 증가, XP 젬 생성, 파티클 생성, 배열 제거까지 처리하는 정리(sweep) 단계 추가. 어떤 무기가 마무리타격을 했는지와 무관하게 매 프레임 시작 시 죽은 몬스터를 확실히 청소함. 발사체/공격 판정에서 이미 처리한 킬은 그 시점에 즉시 배열에서 제거되므로 이 새 로직과 중복 카운트되지 않음.
- **검증**: Node로 죽은(hp<=0) 엔티티와 살아있는 엔티티가 섞인 배열에서 정리 로직만 발췌해 실행 → 죽은 것만 정확히 1건 제거되고 킬 카운트/XP 젬이 생성되는 것을 확인.
